# Лабораторная работа №5. Средства управления проектами.

## 1. Создание и настройка проекта.
В репозитории GitHub был создан новый проект Coursework в виде доски канбан во вкладке Projects.
### В качестве этапов задач были использованы: 
- __Done__ - согласованная задача;
- __In review__ - задача, которую необходимо согласовать;
- __In progress__ - задача в работе;
- __Ready__ - сформулированная и готовая к выполнению задача;
- __Backlog__ - задача на неопределенный срок, рекомендация.
- __Back-to-dev__ - задачу необходимо доработать.

### Добавление кастомных полей.
Из главного окна проекта необходимо зайти по следующему пути: _View more options -> Settings -> Custom fields_.
Далее для удобной маркировки задач мною были добавлены следующие поля:
- __Приоритет__: как можно скорее, средний, несрочно;
- Дата начала/Дата окончания;
- __Тип работы__: исследовательская работа, составление документации, программирование;
- __Результат__: фрагмент кода, фрагмент отчета, готовый алгоритм, отчет.

## 2. Планирование работа по курсовой работе.
### Процесс выполнения курсовой работы был разбит на следующие этапы:  
1. Изучить требования к курсовой работе.
2. Изучение существующих подходов к решению.
3. Изучить способы реализации алгоритма.
4. Реализовать алгоритм на С++.
5. Развертывание проекта.
6. Изучить требования к написанию отчета.
7. Выбрать редактор для написания отчета.
8. Написать отчет.  

Для каждой задачи были выбраны соответствующие ей поля из предыдущего пункта.  
На некоторые тикеты  стоит блокировка: зависимая задача не может быть выполнена, пока не выполнена основная задача. Например, отчет не может быть написан, пока не выбрать текстовый редактор.

Был создан Markdown файл с кратким обзором алгоритма для выполнения лабораторной работы, который привязан к issue. Во время выполнения задания открыт черновой Pull Request, который позже переведен в готовый. При утверждении алгоритма тикет будет перемещен в другое поле этапа выполнения курсовой работы.

## 3. Альтернативы.
Существует множество альтернатив для управления проектами в команде. Например, __Notion__, __Jira__, __Asana__, __YouGile__.  
### На примере YouGile рассмотрим способы управления проектами:
#### Общая информация
__YouGile__ — это российская система управления проектами, которая объединяет в себе функциональность таск-трекера и корпоративного мессенджера.  
YouGile имеет множество функций, характерных для GitHub Projects, такие как описание задач, поля, теги, задание связей между задачами, доски различных форматов (канбан, диаграмма Ганта, календарь и т.д.). Также есть встроенный таск-трекер, который позволяет отслеживать актуальные задачи.  
В YouGile каждая карточка задачи — это отдельный чат, где участники могут обсуждать детали, обмениваться файлами и записывать экран. Встроенный помощник на базе ИИ может создавать проекты по голосовой команде, анализировать обсуждения в задачах, собирать отчёты и управлять сроками.
Связи между задачами в YouGile можно задавать через создание подзадач или через комментарии с упоминанием или ссылкой на связанную задачу.

#### Тарифный план
Бесплатное использование YouGile возможно для команд до  10 человек. Если команда больше, то стандартный платный тариф составляет 495 р/мес за пользователя.

#### Выполнение заданий
Аналогично GitHub Projects, можно создать задачу на YouGile, добавив теги и тип работы, добавить дедлайн, а также назначить ответственного. 
Так как в YouGile нельзя привязать Pull Request, то выполнение задачи может происходить следующим образом: выполнить работу локально, загрузить результат как вложение в задачу или как ссылку, переместить задачу на доске из колонки «В работе» в колонку «На проверке» и в чате задачи упомянуть ревьюера, после принятия — переместить задачу в колонку «Готово».
